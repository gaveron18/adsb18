и MSG,4 (speed) — без координат. `process_snapshot()` был написан правильно
(проверял lat is not None), enqueue() — нет.


но в positions пишется только когда есть реальные координаты.

### 2. Удалено 232578 NULL строк
```sql

VACUUM ANALYZE positions_2026_03;
```

### 3. adsb18-ingest замаскирован
```bash

   /etc/systemd/system/adsb18-ingest.service.disabled
ln -s /dev/null /etc/systemd/system/adsb18-ingest.service

```
Статус: `Loaded: masked` — невозможно запустить ни вручную, ни при ребуте.

### 4. Поллер перезапущен с новым кодом

## Реализация bulk-эндпоинта и оптимизация загрузки архива

### Проблема
Кнопка "Все" в архиве грузила 164 рейса за 18 секунд (у Андрея).

### Диагностика через Playwright

Замеры показали:
- JS-выполнение: ~2 секунды (fetch + parse + addTo map)
- Реальное время у пользователя: 18-21 секунда
- Причина разницы: размер ответа 8.5MB при канале ~0.5MB/s до VPS = ~17 секунд только на скачивание

### Что сделали

**1. POST /api/history/bulk (коммит 4d780c1)**

Новый эндпоинт в server/api/main.py:
- Принимает список рейсов (icao + from_ts + to_ts)
- Одно соединение к БД, последовательные запросы
- Возвращает все треки в одном HTTP-ответе
- До: 164 параллельных запросов при pool=10 → очередь
- После: 1 HTTP-запрос

**2. Переписан selectVisible() в archive.html (коммит 4d780c1)**

Вместо Promise.all(164 fetch) — один POST /api/history/bulk.
Треки появляются все сразу после получения ответа.

**3. Снижен limit_per_flight 300 → 50 (коммит 36a6694)**

| Метрика | Было | Стало |
|---------|------|-------|
| Размер ответа | 8.5 MB | 1.6 MB |
| Точек всего | 40 978 | 7 784 |
| Время у Андрея (~0.5MB/s) | ~18 с | ~3-4 с |

50 точек на рейс достаточно для обзора всех треков на карте.
При клике на конкретный рейс — грузится через /api/history с limit=10000 (полный трек).

### Текущее состояние
- archive.html: selectVisible() использует bulk-эндпоинт
- API: /api/history/bulk готов, limit_per_flight макс 2000
