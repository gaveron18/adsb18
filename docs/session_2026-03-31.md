# Session 2026-03-31 — Диагностика архива, чистка NULL позиций, маскировка ingest

## Что обнаружили

### Проблема: архив не показывал треки (жалоба пользователя)
- Пользователь видел трек над Ижевском час назад, потом он "пропал"
- API и БД работали нормально — NWS744 (151E62) в архиве с 2544 точками
- Реальная причина: 87% записей в positions имели lat=NULL

### Диагностика показала:
- `adsb18-ingest.service` работал 13 часов (с 30.03 21:43) несмотря на disabled
- За последний час: 24284 записей в positions, из них только 3164 (13%) с координатами
- 232578 строк с lat=NULL засоряли таблицу positions_2026_03

### Корневая причина бага:
`enqueue()` в db.py добавлял в _batch ВСЕ SBS-сообщения включая MSG,1 (callsign)
и MSG,4 (speed) — без координат. `process_snapshot()` был написан правильно
(проверял lat is not None), enqueue() — нет.

## Что сделали

### 1. Фикс db.py — enqueue() (коммит 2612c31)
Добавлена проверка перед добавлением в _batch:
```python
if s.get(lat) is None or s.get(lon) is None:
    return
```
_state по-прежнему обновляется для всех сообщений (callsign, speed и т.д.),
но в positions пишется только когда есть реальные координаты.

### 2. Удалено 232578 NULL строк
```sql
DELETE FROM positions_2026_03 WHERE lat IS NULL;
VACUUM ANALYZE positions_2026_03;
```

### 3. adsb18-ingest замаскирован
```bash
mv /etc/systemd/system/adsb18-ingest.service \
   /etc/systemd/system/adsb18-ingest.service.disabled
ln -s /dev/null /etc/systemd/system/adsb18-ingest.service
systemctl daemon-reload
```
Статус: `Loaded: masked` — невозможно запустить ни вручную, ни при ребуте.

### 4. Поллер перезапущен с новым кодом
adsb18-poller.service перезапущен. Проверка: 1035/1035 с lat за последние 30 мин = 100%.

## Текущее состояние
- positions: только строки с lat/lon (100%)
- adsb18-ingest: masked → /dev/null
- adsb18-poller: running, enabled — единственный источник данных

## Инцидент: зависание браузера после перезапуска API

### Что случилось
- Перезапуск `adsb18-api` создал ~3-секундное окно с 502 ошибками
- Браузер Андрея держал keep-alive соединения и получил 502
- После подъёма API данные встали в TCP send buffer (106K байт не уходили)
- Диагноз: `ss -tnp | grep 8098` → `ESTAB send=106203 → 81.4.255.145`

### Решение
```bash
sudo nginx -s reload   # сбросить зависшие соединения
```
Затем Ctrl+Shift+R в браузере.

### Правило на будущее
После `systemctl restart adsb18-api` — сразу `sudo nginx -s reload`.
