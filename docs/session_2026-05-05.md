## Гипотезы (по убыванию правдоподобия)

1. **PBD1569 сегодня не передаёт DF17 (Position) в нашу сторону** — самая
   правдоподобная: при 7772 пакетах за 30 минут и rssi −2.5 dBm физически

   у PBD), но 09:53 и 09:59 при таких же или более сильных сигналах декодирование
   у других бортов работало (cpr_ok=79..95). Значит overload — не основная

### 100% доказательство по DF17 Type Code

DF17 (длинный 112 µs) содержит разные подтипы:
- TC=4   Aircraft Identification (callsign)
- TC=9-18 Airborne Position ← ADS-B координаты
- TC=19  Airborne Velocity (gs/heading/baro_rate)
- TC=29  Target State (nav_qnh/nav_alt/nav_heading)
- TC=31  Operational Status (version/nac_p/sil)

Все TC = в одном DF17 одинаковой длины и одинаково подвержены помехе.
Если приёмник ловит TC=19/29/31, то и TC=9-18 он бы ловил тоже.

Снимок 13 бортов в эфире (09:50 МСК):

| ICAO | rssi | TC1-4 | TC9-18 | TC19 | TC29 | TC31 |
|---|---|---|---|---|---|---|
| PBD570 (151e4f) | −4.2 | ✓ | ✗ | ✗ | ✓ | ✓ |
| SDM6591 | −5.0 | ✓ | ✗ | ✓ | ✗ | ✓ |
| SVR503 | −5.2 | ✓ | ✗ | ✓ | ✓ | ✓ |
| AFL1460 | −17.5 | ✓ | ✗ | ✓ | ✓ | ✓ |
| AFL1201 | −24.1 | ✓ | ✗ | ✓ | ✓ | ✓ |
| ETH3499 | −33.0 | ✓ | (история) | ✓ | ✓ | ✓ |

У всех российских (PBD/AFL/SVR/SDM): мы ловим **TC=19/29/31 — длинные
DF17**, но **TC=9-18 (Position) нет**. Тракт декодирует длинные DF17 этих
бортов чисто. У ETH3499 — есть pos=+ при rssi −33, тракт работает.

NAC_p из TC=31 у российских = 0 (декларируемая нулевая точность позиции
самим бортом), у ETH3499 = 1. Это **самим бортом** сообщаемое значение,
не наш приёмник.

**Вывод:** российские транспондеры сейчас не передают Position-фрейм
(или передают без валидной точности → readsb отбрасывает). Это совпадает
с известной практикой РФ с 2022 года из-за рисков GPS-спуфинга.

### CCA936 — «обрыв на 60 км»

Андрей видит в archive.html на adsb18.ru обрыв трека CCA936 на ~60 км
от Pi.

Координаты трека (icao 781CE6):

```
ПЕРВАЯ точка    23:44:12  lat 58.47  lon 51.30  d_pi=278  heading 105°  ← запад
ЗЕНИТ           00:01:51  lat 57.98  lon 55.93  d_pi=2.9                ← над Pi
ПОСЛЕДНЯЯ       00:20:58  lat 57.51  lon 60.92  d_pi=298  heading  93°  ← восток
```

Это **транзит Air China на 39000 ft** Запад→Восток, не садился в Б.Савино.
В зоне 60 км от Pi: 405 точек за 456 сек, **максимальный гэп 3 секунды**,
гэпов > 5 сек = 0. То есть трек в зоне Pi/Савино **целостный**.

Все большие гэпы трека (244, 59, 40 сек) — на расстоянии 150–290 км
от Pi (естественный радиогоризонт, борт за горизонтом).

### Корневая причина видимого «обрыва на 60 км» — баг в archive.html

`frontend/archive.html` строка 892:
```javascript
const url = `/api/history?icao=${f.icao}&from=...&to=...&limit=500`;
                                                       ^^^^^^^^^
```

API при `limit=500` возвращает **первые 500 точек по времени** (default
у API = 2000, max = 10000). Для CCA936 (1419 точек) первые 500 покрывают
период 23:44 → 23:59 МСК, последняя из них на **39.9 км** от Pi.

Дальнейшие 919 точек (зенит 2.9 км → отлёт 297 км) фронтенд **не запрашивает**
и не показывает.

Эмпирическая проверка через curl на разных limit:

```
limit=100  →  100 точек,  последняя на 157.0 км
limit=500  →  500 точек,  последняя на  39.9 км ← то что видит Андрей
limit=1000 → 1000 точек,  последняя на 107.3 км
limit=2000 → 1419 точек,  последняя на 301.7 км ← полный трек
```

«60 км» Андрея = ~40 км фактически (примерно на глаз). Фикс — изменить
`limit=500` на `limit=10000` в строке 892 archive.html.

### Этап 2 — диагностика: почему Pi-туннель к dev не работает

`adsb-tunnel.service` на Pi (autossh → `new@173.249.2.184`) — active,
но за сутки 800+ рестартов с `Connection to 173.249.2.184 port 22
timed out`. На dev `:30092` не слушает; dev poller получает
`Connection refused` каждый poll, БД dev замёрзла на `2026-04-30`.

`adsb-tunnel-prod.service` (тот же autossh, тот же ключ, тот же ПО, но
к prod) — **работает идеально**, держится сутками.

Корневая причина (доказана эмпирически с обоих концов через
ProxyJump-доступ к Pi):

| Тест | Результат |
|------|-----------|
| Whois Pi public IP `89.253.38.94` | T2 Russia AS48190, EKB-сегмент (мобильный оператор) |
| Pi → dev `ping -M do -s 1472` | 0% loss (PMTUD не виноват) |
| Pi → dev `nc :22` | succeeded (TCP открывается) |
| Pi → dev `ssh -vvv` ручной | KEX **проходит** до `SSH2_MSG_NEWKEYS` за <8 сек |
| Pi → dev `autossh` в фоне 90-сек циклом | **Connection timed out** (систематически) |
| Pi → dev UDP `:51820` (WireGuard) | **0 байт обратно** (полный блок) |

Гипотеза Андрея подтверждена с уточнением: мобильный DPI Tele2
полностью режет UDP-VPN к зарубежным IP, и **прерывисто** режет SSH
к зарубежному IP. Ручной ssh иногда успевает, autossh с retry-периодом
90 сек систематически попадает в окно блокировки. prod (РФ-IP) не
страдает — оператор не вмешивается.

ТСПУ-DPI на маршрут **prod (РФ-ЦОД) → dev (DE-ЦОД)** не активен (тест:
`ssh -vv root@prod 'ssh new@dev'` → KEX OK, упёрлось в Permission denied).
