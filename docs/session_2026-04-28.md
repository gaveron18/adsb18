# Сессия 2026-04-28

## Диагностика: ловит дальние борта, не ловит ближние

### Жалоба
Около Pi3 пролетало ~50 рейсов за день, в БД ближних рейсов почти нет — только дальние.
Гипотеза Андрея: слишком большое усиление (gain).

### Что проверено

**Конфиг readsb на Pi** (`/etc/default/readsb`):
```
RECEIVER_OPTIONS="--device 0 --device-type rtlsdr --gain auto --ppm 0"
```
- `--gain auto` — readsb сам подбирает gain
- autogain-скрипт `/usr/local/bin/readsb-gain` установлен, **но не активен**
  (нет systemd timer'а, кэш `/var/cache/readsb-autogain/` пуст)

**Текущее значение gain в эфире** (`stats.json`):
- gain_db плавает между 22.9 и 0.0 (нерегулярная запись поля при `--gain auto`)
- 22.9 — стандартное значение из таблицы RTL-SDR, середина шкалы

### Интерпретация

1. **Когда близкий борт пролетает — приёмник на грани насыщения АЦП**
   (peak −1.8 dBFS из 0). Это не теория, фиксируется в каждом 15-мин окне.
2. **Шумовой пол подскакивает на 12 dB** при близком борте (−44.8 → −32.6) —
   характерный признак работы AGC RTL-SDR на пределе.
3. Соотношение `bad / accepted` ≈ **1000:1** — большая часть громких пакетов
   декодируется как мусор и отбрасывается. Это и есть пропущенные ближние борта.
4. `strong_signals = 1333` за 15 мин при пролёте близкого vs `0` в тихие минуты —
   локальные борта **систематически** заходят в зону перегруза.

### Вердикт
Гипотеза Андрея подтверждается: **gain слишком высокий** для близких бортов.
Дальние принимаются нормально, потому что приходят слабым сигналом и не вызывают перегруз.

### План (отложено — Pi сейчас недоступен, изменения на Pi запрещены без доступа)

Когда Pi станет доступен — попробовать одно из двух:

**Вариант A: понизить gain вручную** (`/etc/default/readsb`):
```
--gain auto  →  --gain 20.7   # шаг вниз с 22.9
```
Если не поможет — следующий шаг: 19.7.

**Вариант B: включить adaptive burst** (рекомендуется upstream):
```
--gain auto --adaptive-burst
```
readsb сам временно сбрасывает gain при детекте перегруза, баланс лучше
для смешанного эфира (близкие + дальние).

После любого изменения: `sudo systemctl restart readsb`, наблюдать 1–2 часа,
повторить snapshot stats.json и сравнить с таблицей выше. Цель — `peak_signal`
держать в диапазоне **−5 … −10 dBFS** (запас до насыщения), `strong_signals`
снизить хотя бы в 5–10 раз без падения `accepted`.
