# Сессия 2026-04-07

## Фича: активная сессия в логе приёмника (коммит 61605d8)

### Проблема
Окно «Лог работы приёмника» не показывало текущую активную сессию Pi.
При открытии модала период брался из архивных дат (которые могли быть за прошлые дни),
и текущая сессия не попадала в запрос.

### Что сделано

**API (`server/api/main.py` — `/api/receiver-log`):**
- Добавлена проверка `pi_active`: если `max(ts)` в positions < 5 мин назад — Pi активен
- Последняя сессия в выборке получает `is_active: true` и `end_utc = now`
- Условие активности: `pi_active AND последняя сессия AND (now - end) < 10 min`
  — третье условие нужно чтобы старые сессии не помечались активными при фильтре по прошлым датам

**Frontend (`frontend/archive.html`):**
- CSS: `.rl-active td` — зелёный фон (#e8f8ee), зелёный текст (#1a7a3a)
- CSS: `.rl-active-badge` — значок «● работает»
- `openReceiverLog`: `rlTo` всегда расширяется до сегодня — активная сессия видна
  даже если архив открыт на прошлый день
- `renderRlTable`: активная строка — зелёная, время окончания «сейчас ►»

### Баг найден и исправлен в процессе тестирования
При фильтре по старым датам (например апрель 1-3) последняя сессия в выборке
помечалась активной — так как `pi_active=True` и `i == len(sessions)-1`.
Исправлено: добавлена проверка `(now - end) < timedelta(minutes=10)`.

### Тесты пройдены
- Старые даты → active: 0 ✓
- Сегодня → active: 1 ✓  
- Запрос без фильтра (30 дней) → active: 1 ✓
- Длительность активной сессии считается корректно ✓

## Фикс: receiver-log — Pi активен без самолётов (коммит 65f1d3e)

### Проблема
После предыдущей сессии выяснилось: если Pi работает, но самолётов в зоне нет,
окно лога не показывало активную сессию. pi_active = False, хотя Pi онлайн.

### Причина
pi_active проверял max(ts) FROM positions < 5 мин.
Без самолётов позиции не пишутся — last_pos_ts устарел (был 43 мин назад).

### Что сделано
- Добавлена _check_pi_alive() — curl к :30092, проверяет поле now < 30с
- pi_active = await _check_pi_alive() вместо проверки БД
- Убрана проверка (now - end) < 10 мин — не нужна при туннельной проверке

### Тесты
- Pi онлайн, самолётов нет — is_active: True
- API возвращает активную сессию сегодняшнего дня

## Удаление adsb18-feeder (план, статус: в работе)

### Почему удаляем
`adsb18-feeder.service` на Pi подключается на порт 30091 -> через -L туннель -> VPS:30001 (adsb18-ingest).
Но `adsb18-ingest` замаскирован в /dev/null с 31 марта (переход на poller-архитектуру).
Фидер работает вхолостую — данные уходят в никуда.
Реальный поток данных: Pi aircraft.json -> HTTP туннель :30092 -> VPS poller -> DB.

### Состояние до изменений
- Pi: adsb18-feeder.service — active (running)
- Pi: adsb-tunnel.service — содержит строку: -L 30091:localhost:30001
- Репо: feeder/update_pi.sh — синхронизирует feeder.py + adsb18-feeder.service (оба не нужны)
- Репо: CLAUDE.md — таблица сервисов содержит adsb18-feeder

### План действий (по шагам)

Шаг 1 — Pi: остановить и удалить adsb18-feeder.service
  sudo systemctl stop adsb18-feeder
  sudo systemctl disable adsb18-feeder
  sudo rm /etc/systemd/system/adsb18-feeder.service
  sudo systemctl daemon-reload

Шаг 2 — Pi: убрать -L из adsb-tunnel.service
  Строка -L 30091:localhost:30001 нужна только для фидера — убрать.
  Перезапуск туннеля: SSH отвалится на ~15 сек, потом восстановится.

Шаг 3 — Репо: переписать feeder/update_pi.sh
  Убрать: FEEDER_NAME, feeder.py, adsb18-feeder.service.
  Оставить: только синхронизацию adsb-tunnel.service.
  feeder.py и feeder_json.py в репо не трогать (архив).

Шаг 4 — Репо: обновить CLAUDE.md
  Таблица сервисов: убрать строку adsb18-feeder.

Шаг 5 — Обновить TROUBLESHOOTING.md
  Записать что adsb18-feeder удалён и почему.

Шаг 6 — git push в одном коммите
  Все изменения репо (update_pi.sh + CLAUDE.md + TROUBLESHOOTING.md + session лог).

### Как откатить

Восстановить adsb18-feeder.service на Pi (содержимое файла):
  /etc/systemd/system/adsb18-feeder.service
  [Unit]
  Description=adsb18 ADS-B Feeder
  After=network-online.target adsb-tunnel.service
  Wants=network-online.target
  [Service]
  ExecStart=/usr/bin/python3 /opt/adsb18-feeder/feeder.py --server 127.0.0.1 --port 30091 --name ads-b-pi --buffer /opt/adsb18-feeder/feeder_buffer.sbs
  Restart=always
  RestartSec=10
  User=ads-b
  [Install]
  WantedBy=multi-user.target
  
  sudo systemctl daemon-reload && sudo systemctl enable adsb18-feeder && sudo systemctl start adsb18-feeder

Восстановить -L в adsb-tunnel.service:
  Добавить строку "  -L 30091:localhost:30001 \" перед строкой "  new@173.249.2.184"
  sudo systemctl restart adsb-tunnel
